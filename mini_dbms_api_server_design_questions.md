## 5. 스레드 풀 자체에 대해 던질 질문

- worker thread는 몇 개 둘 것인가? 8개
  > 스레드 풀 크기 결정 기준

    스레드 풀의 worker thread 개수는 우선 CPU 코어 수를 기준으로 8로 설정한다. 

    그 이유는, 특정 시점에 실제로 동시에 실행될 수 있는 스레드 수가 결국 머신의 CPU 병렬 처리 능력에 의해 제한되기 때문이다. 물론 서버 프로세스가 시스템의 모든 코어를 독점한다고 볼 수는 없지만, 코어 수는 해당 환경이 제공할 수 있는 물리적인 병렬 실행 능력의 가장 직관적인 1차 기준이 된다.

    또한 worker thread를 과도하게 많이 생성한다고 해서 성능이 반드시 좋아지는 것은 아니다. 오히려 스레드 수가 지나치게 많아지면 다음과 같은 문제가 생길 수 있다.

    컨텍스트 스위칭 비용 증가
    CPU 캐시 locality 저하
    요청 큐 및 락 경쟁 증가
    디버깅과 성능 분석의 복잡도 증가

    특히 이번 과제의 API 서버는 단순한 네트워크 I/O만 수행하는 구조가 아니라, 요청 처리 과정에서 SQL 파싱, DB 엔진 실행, B+트리 탐색, 락 획득, 파일 접근 가능성이 함께 존재한다. 따라서 이 서버를 순수한 I/O-bound 서버로 단정하기 어렵고, 상황에 따라 CPU 연산 비용이나 락 대기 비용이 더 크게 작용할 수 있다. 이런 조건에서는 worker thread를 무작정 많이 두기보다, 코어 수 근처의 고정 개수로 시작하는 것이 더 안정적이고 합리적이다.

    즉, 이번 설계에서 코어 수를 기준으로 삼은 이유는 “이 프로세스가 모든 코어를 점유할 수 있기 때문”이 아니라, 과도한 oversubscription을 피하면서 합리적인 초기 병렬성 수준을 정하기 위한 휴리스틱으로서 가장 적절하기 때문이다.

    최종적으로는 CPU 코어 수를 기준으로 초기값을 정한 뒤, 이후 벤치마크를 통해 throughput과 latency를 측정하면서 필요한 경우 조정한다.
- 요청 큐는 bounded queue로 둘 것인가? 응, 32개 요청까지 대기
  - 요청 큐의 크기는 서버의 가용 메모리뿐 아니라, worker thread 수, 요청 처리 시간, 허용 가능한 대기 시간까지 함께 고려하여 결정한다. 큐가 지나치게 작으면 순간적인 요청 증가도 흡수하지 못하고, 반대로 지나치게 크면 메모리 사용량 증가와 과도한 대기 시간으로 인해 오히려 시스템 응답성이 악화될 수 있다. 따라서 초기 구현에서는 worker thread 수를 기준으로 일정 배수(초기값 4)의 bounded queue를 두고, 이후 부하 테스트 결과에 따라 조정한다.
- 큐가 가득 차면 어떻게 처리할 것인가?
  - 큐 포화 상황에서는 fail-fast 정책을 적용한다. 이미 worker와 대기열이 모두 가득 찬 상태에서 추가 요청까지 계속 수용하면, 메모리 사용량 증가와 응답 지연 악화로 인해 시스템 전체 안정성이 떨어질 수 있다. 따라서 서버는 초과 요청을 즉시 거절하고, overload 상태를 명시적으로 드러내는 방향으로 설계한다.
- 스레드가 직접 소켓 응답까지 보낼 것인가, 아니면 결과만 넘길 것인가?
  - 본 설계에서는 worker thread가 요청 처리뿐 아니라 소켓 응답 전송까지 직접 수행하도록 한다. 즉, 하나의 요청은 하나의 worker가 큐에서 꺼낸 뒤, parse → execute → respond의 전 과정을 끝까지 담당한다. 이는 요청 처리 흐름을 단순하게 유지하고, 중간 결과를 다른 스레드에 전달하기 위한 추가 큐나 동기화 구조를 줄이기 위한 선택이다.
  본 설계에서 worker thread가 응답 전송까지 직접 담당하는 구조는 구현이 단순하다는 장점이 있지만, 동시에 몇 가지 한계도 가진다. 대표적으로 worker가 소켓 write 과정에서 block될 수 있으며, 클라이언트 수신 속도가 느린 경우 DB 실행은 이미 끝났더라도 worker가 반환되지 않아 전체 처리량이 떨어질 수 있다. 또한 처리 로직과 전송 로직이 분리되지 않기 때문에, 대규모 응답이나 고성능 non-blocking I/O 구조로 확장할 때는 비효율적일 수 있다.
  다만 본 과제에서는 이러한 단점이 크게 드러날 가능성이 낮다고 판단했다. 그 이유는 API 응답 크기가 비교적 작고, 테스트 환경이 통제된 로컬 또는 단순 클라이언트 중심일 가능성이 높으며, 과제의 핵심 목표가 네트워크 I/O 최적화보다는 스레드 풀 기반 요청 처리와 DB 엔진 연동 구조를 구현하는 데 있기 때문이다. 따라서 본 구현에서는 구조적 단순성과 디버깅 용이성을 우선하여, worker thread가 요청 처리부터 응답 전송까지 전 과정을 직접 담당하도록 설계한다.
- 디버깅과 구현 단순성을 위해 가장 단순한 모델은 무엇인가?

## 6. API 모양을 정할 때 던질 질문

- API는 REST처럼 보이게 만들 것인가, SQL gateway처럼 만들 것인가?

  - REST 스타일 API는 자원 중심의 endpoint를 정의하여 외부에 일반적인 웹 API처럼 보이게 만드는 방식이고, SQL gateway 스타일 API는 SQL 문자열 자체를 요청으로 받아 내부 SQL 처리기에 전달하는 방식이다. 전자는 외부 인터페이스가 직관적이지만, 기존 SQL 처리기의 재사용성이 떨어지고 endpoint 설계가 복잡해질 수 있다. 반면 후자는 POST /query 같은 단일 endpoint만으로 기존 SQL 엔진을 직접 활용할 수 있어 구현이 단순하다.
- 가장 단순한 요청/응답 포맷은 무엇인가?
  - 가장 단순한 요청/응답 포맷은, 요청에서 실행할 SQL 문자열을 명시적으로 전달하고 응답에서 성공 여부와 실행 결과를 구조적으로 반환할 수 있는 형태여야 한다. 이를 위해 요청은 JSON body로 { "sql": "..." } 형식을 사용하고, 응답 역시 JSON으로 success, rows, error 등의 필드를 포함하도록 설계하는 것이 가장 단순하고 실용적이다. 이 방식은 기존 SQL 처리기에 쉽게 연결할 수 있고, 클라이언트 측에서도 성공/실패 분기와 결과 처리가 명확하다는 장점이 있다.

        요청
      {
        "sql": "SELECT * FROM users WHERE id = 1;"
      }
      성공 응답
      {
        "success": true,
        "rows": [
          { "id": 1, "name": "Alice" }
        ]
      }
      실패 응답
      {
        "success": false,
        "error": "syntax error near FROM"
      }

## 12. 현실적인 MVP 방향 예시

- API: `POST /query`
- 요청 body: `{ "sql": "..." }`
- main thread는 `accept + queue push`
- worker thread가 요청을 꺼내 처리
- worker가 SQL 실행 후 JSON 응답 반환
- 동시성 제어:
  - DB 전체에 `pthread_rwlock_t`
  - `SELECT`는 read lock
  - `INSERT/UPDATE/DELETE/CREATE`는 write lock
- 저장:
  - 데이터 파일 저장
  - 인덱스는 시작 시 rebuild
- 테스트:
  - engine unit test
  - API integration test
  - concurrent request test
- 차별점:
  - `execution_time_ms`
  - `index_used: true/false`
  - `/health`, `/metrics`
